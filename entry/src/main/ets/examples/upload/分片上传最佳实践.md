# 分片上传最佳实践指南

## 1. 概述

分片上传（Chunked Upload）是一种将大文件分割成多个小块进行上传的技术。这种方式可以有效解决大文件上传时的网络超时、内存占用过高等问题，并支持断点续传功能。

本示例展示了如何使用 WebServer 框架实现一个完整的分片上传系统。

## 2. 核心功能

- **分片上传**: 将大文件分割成多个小块，逐个上传到服务器
- **断点续传**: 支持上传中断后从断点处继续上传
- **进度跟踪**: 实时跟踪上传进度
- **文件校验**: 使用文件哈希值确保文件完整性
- **任务管理**: 支持查询、取消上传任务
- **自动合并**: 所有分片上传完成后自动合并成完整文件

## 3. 工作流程

### 3.1. 客户端流程

```
1. 选择文件
   ↓
2. 计算文件哈希值（用于唯一标识文件）
   ↓
3. 检查服务器是否已有该文件或部分分片
   ↓
4. 将文件分割成固定大小的分片
   ↓
5. 逐个上传分片（跳过已上传的分片）
   ↓
6. 所有分片上传完成后，请求服务器合并
   ↓
7. 服务器返回完整文件信息
```

### 3.2. 服务器流程

```
接收分片请求
   ↓
验证分片参数（索引、总数、哈希值等）
   ↓
保存分片到临时目录
   ↓
更新上传任务状态
   ↓
返回上传进度
   ↓
接收合并请求
   ↓
验证所有分片是否完整
   ↓
按顺序合并分片
   ↓
清理临时文件
   ↓
返回完整文件信息
```

## 4. API 接口说明

### 4.1. 检查上传状态

**接口**: `POST /api/upload/check`

**功能**: 检查文件是否已存在或部分上传，用于实现断点续传

**请求体**:
```json
{
  "fileHash": "文件的哈希值",
  "fileName": "文件名"
}
```

**响应**:
```json
{
  "exists": true,
  "uploadedChunks": [0, 1, 2],  // 已上传的分片索引
  "totalChunks": 10
}
```

### 4.2. 上传分片

**接口**: `POST /api/upload/chunk`

**功能**: 上传单个文件分片

**请求**: `multipart/form-data`
- `chunk`: 分片文件（二进制数据）
- `chunkIndex`: 分片索引（从0开始）
- `totalChunks`: 总分片数
- `fileName`: 文件名
- `fileHash`: 文件哈希值

**响应**:
```json
{
  "message": "Chunk uploaded successfully",
  "chunkIndex": 0,
  "uploadedChunks": 1,
  "totalChunks": 10,
  "progress": "10.00"
}
```

### 4.3. 合并分片

**接口**: `POST /api/upload/merge`

**功能**: 将所有分片合并成完整文件

**请求体**:
```json
{
  "fileHash": "文件的哈希值",
  "fileName": "文件名"
}
```

**响应**:
```json
{
  "message": "File merged successfully",
  "file": {
    "name": "test.bin",
    "size": 10485760,
    "path": "/path/to/file",
    "hash": "abc123...",
    "uploadTime": "2025-12-31T10:00:00.000Z"
  }
}
```

### 4.4. 取消上传

**接口**: `DELETE /api/upload/cancel/:fileHash`

**功能**: 取消上传任务并清理临时文件

**响应**:
```json
{
  "message": "Upload cancelled successfully",
  "fileHash": "abc123..."
}
```

### 4.5. 查询上传任务

**接口**: `GET /api/upload/tasks`

**功能**: 获取所有进行中的上传任务

**响应**:
```json
{
  "tasks": [
    {
      "fileHash": "abc123...",
      "fileName": "test.bin",
      "totalChunks": 10,
      "uploadedChunks": 5,
```

### 4.6. 查询已上传文件

**接口**: `GET /api/upload/files`

**功能**: 获取所有已上传完成的文件列表

**响应**:

### 5.3. 客户端实现（JavaScript）

```javascript
// 1. 计算文件哈希（简化版，实际应使用更可靠的哈希算法）
async function calculateFileHash(file) {
  const buffer = await file.arrayBuffer();
  const hashBuffer = await crypto.subtle.digest('SHA-256', buffer);
  const hashArray = Array.from(new Uint8Array(hashBuffer));
  return hashArray.map(b => b.toString(16).padStart(2, '0')).join('');
}

// 2. 检查上传状态
async function checkUploadStatus(fileHash, fileName) {
  const response = await fetch('http://server:8088/api/upload/check', {
    method: 'POST',
    headers: { 'Content-Type': 'application/json' },
    body: JSON.stringify({ fileHash, fileName })
  });
  return await response.json();
}

// 3. 上传分片
async function uploadChunk(chunk, chunkIndex, totalChunks, fileName, fileHash) {
  const formData = new FormData();
  formData.append('chunk', chunk);
  formData.append('chunkIndex', chunkIndex);
  formData.append('totalChunks', totalChunks);
  formData.append('fileName', fileName);
  formData.append('fileHash', fileHash);

  const response = await fetch('http://server:8088/api/upload/chunk', {
    method: 'POST',
    body: formData
  });
  return await response.json();
}

// 4. 合并文件
async function mergeFile(fileHash, fileName) {
  const response = await fetch('http://server:8088/api/upload/merge', {
    method: 'POST',
    headers: { 'Content-Type': 'application/json' },
    body: JSON.stringify({ fileHash, fileName })
  });
  return await response.json();
}

// 5. 完整上传流程
async function uploadFile(file) {
  const CHUNK_SIZE = 1024 * 1024; // 1MB
  const fileName = file.name;
  const fileHash = await calculateFileHash(file);
  const totalChunks = Math.ceil(file.size / CHUNK_SIZE);

  // 检查已上传的分片
  const status = await checkUploadStatus(fileHash, fileName);
  const uploadedChunks = new Set(status.uploadedChunks || []);

  // 上传分片
  for (let i = 0; i < totalChunks; i++) {
    if (uploadedChunks.has(i)) {
      console.log(`分片 ${i} 已存在，跳过`);
      continue;
    }

    const start = i * CHUNK_SIZE;
    const end = Math.min(start + CHUNK_SIZE, file.size);
    const chunk = file.slice(start, end);

    const result = await uploadChunk(chunk, i, totalChunks, fileName, fileHash);
    console.log(`上传进度: ${result.progress}%`);
  }

  // 合并文件
  const mergeResult = await mergeFile(fileHash, fileName);
  console.log('上传完成:', mergeResult);
}
```

## 7. 最佳实践建议

### 7.1. 分片大小选择

- **小文件（< 10MB）**: 不建议使用分片上传
- **中等文件（10MB - 100MB）**: 建议分片大小 1-2MB
- **大文件（> 100MB）**: 建议分片大小 2-5MB
- **超大文件（> 1GB）**: 建议分片大小 5-10MB

### 7.2. 错误处理

- 实现重试机制：单个分片上传失败时自动重试（建议最多3次）
- 超时处理：设置合理的超时时间（建议30-60秒）
- 网络异常：保存上传状态，支持断点续传

### 7.3. 性能优化

- **并发上传**: 可以同时上传多个分片（建议3-5个并发）
- **压缩传输**: 对可压缩的文件类型启用压缩
- **CDN加速**: 使用CDN节点就近上传
- **限流控制**: 服务器端实现上传速率限制

### 7.4. 安全考虑

- **文件类型验证**: 检查文件扩展名和MIME类型
- **文件大小限制**: 设置单个文件和总存储空间限制
- **访问控制**: 实现用户认证和授权机制
- **病毒扫描**: 对上传的文件进行安全扫描
- **哈希校验**: 上传完成后验证文件完整性

### 7.5. 存储管理

- **定期清理**: 清理超时未完成的上传任务
- **存储配额**: 为每个用户设置存储配额
- **文件去重**: 使用哈希值避免重复存储相同文件
- **归档策略**: 对长期未访问的文件进行归档

## 8. 常见问题

### Q1: 如何处理上传中断？

A: 客户端在重新上传时，先调用 `/api/upload/check` 接口查询已上传的分片，然后只上传缺失的分片。

### Q2: 分片顺序必须按序上传吗？

A: 不需要。分片可以乱序上传，服务器会记录每个分片的索引，合并时按索引顺序组装。

### Q3: 如何防止恶意上传？

A: 建议实现以下措施：
- 用户认证和授权
- 上传速率限制
- 文件大小和类型限制
- IP黑名单机制

### Q4: 上传失败如何清理临时文件？

A: 服务器应该实现定时任务，清理超过一定时间（如24小时）未完成的上传任务及其临时文件。

### Q5: 如何提高上传速度？

A: 可以采用以下方法：
- 增加分片大小（减少HTTP请求次数）
- 实现并发上传（同时上传多个分片）
- 使用CDN或就近的服务器节点
- 启用HTTP/2或HTTP/3协议
